# 第一、二阶段性能与稳定性优化执行清单

## 目标

在不改变德州扑克核心规则、现有存档格式与用户体验的前提下，完成两阶段优化：

- 将普通对局与锦标赛对局的主要渲染开销降低，使常规场景尽量稳定在 60 FPS。
- 限制后台模拟、AI、LLM 和锦标赛任务对 UI 主线程的干扰。
- 建立能定位卡顿、内存增长、线程堆积与数据写入问题的可观测性。
- 将关键资金变更从全量 JSON 重写升级为可恢复的追加式账本。
- 为后续 World 2.0 的区域、NPC、经济、任务扩展准备清晰的任务和事件边界。

## 基线与验收口径

### 当前基线

- 普通对局 `playing`：平均约 28.8 FPS，平均渲染约 42.7ms。
- 锦标赛 `tournament`：平均约 39.9 FPS，平均渲染约 27.6ms。
- 低频但严重的帧尖峰：普通对局最大约 1517ms，锦标赛最大约 763ms。
- RSS 在长时间游玩中存在明显上升，需要持续采样确认增长来源。

### 通用验收

每个任务完成后均执行：

1. 静态语法检查：受影响 Python 文件可被 `ast.parse` 解析。
2. 启动检查：游戏可进入菜单、设置、普通对局和锦标赛报名页。
3. 回归检查：普通对局可完成一手、离开后结算；锦标赛可进入结果页。
4. 日志检查：`perf.log` 与审计账本没有异常堆栈、死锁、持续刷错或字段缺失。
5. 任务状态：本文件对应任务从 `[ ]` 改为 `[x]`，并记录实际结果和日志对比。

**状态：** `[ ] 未开始`

# 每轮优化后的游玩验证流程

完成阶段一和阶段二后，使用新包进行至少一次完整游玩并保存 `perf.log`：

1. 菜单停留 2 分钟，切换设置、历史、全屏。
2. 进入 2 人、4 人、8 人普通对局，每种至少 5 手。

5. 完整进行或自动模拟一次锦标赛至结果页。
6. 再次进入菜单和历史页，记录 RSS、线程数、帧尖峰。
7. 保存优化前后日志，比较：平均/最低 FPS、Render 平均/最大值、>100ms 帧次数、RSS 增量、线程峰值、后台吞吐与任务失败数。
